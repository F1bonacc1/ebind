# Milestone v1.0: Pause/Resume DAG Execution

**Status:** ✅ SHIPPED 2026-06-03
**Phases:** 1-4
**Total Plans:** 8

## Overview

Add pause and resume capability to ebind's DAG workflow engine. Users can pause a running DAG (in-flight steps complete naturally, pending steps wait), persist the paused state across restarts, and resume execution later. Zero CPU consumption while paused.

## Phases

### Phase 1: State Machine & Pure Logic

**Goal**: Foundation types and pure state logic for pausing/paused DAG statuses are correct, tested, and consistent with existing patterns.
**Depends on**: Nothing
**Plans**: 1 plan

Plans:

- [x] 01-01-PLAN.md — Add state constants, pure pause/resume functions, update Cancel/maybeFinalize/Terminal, add PausedAt field, write unit tests

**Details:**

- `DAGStatusPausing` ("pausing") and `DAGStatusPaused` ("paused") constants follow existing lowercase naming convention
- `HasInFlightSteps(state)` returns `true` iff at least one step has `StatusRunning`
- `CanPause(meta)` returns `true` only when `meta.Status == DAGStatusRunning`; `CanResume(meta)` returns `true` only when `meta.Status == DAGStatusPaused`
- `maybeFinalize` does not transition pausing or paused DAGs to any final state
- `Cancel()` transitions pausing and paused DAGs to canceled via CAS
- 13 pure unit tests pass with `-race`

### Phase 2: Scheduler Pause Awareness

**Goal**: Scheduler correctly gates event processing, step dispatch, and sweep recovery for pausing/paused DAGs without consuming CPU for idle paused DAGs.
**Depends on**: Phase 1
**Plans**: 2 plans

Plans:

- [x] 02-01-PLAN.md — Add scheduler pause gates and auto-transition (SG-01, SG-02, SG-03, SG-04)
- [x] 02-02-PLAN.md — Write scheduler pause unit tests (TST-02)

**Details:**

- Completion events for steps in pausing/paused DAGs are Acked without triggering step transitions or downstream dispatch (SG-01)
- `enqueueReady` skips steps belonging to pausing or paused DAGs — no new dispatches while dormant (SG-02)
- On leader acquisition, sweep detects pausing DAGs and transitions them to paused; sweep skips paused DAGs entirely (SG-03)
- pausing→paused transition fires automatically when the last in-flight step completes, driven by the event loop without a ticker (SG-04)
- 7 scheduler pause unit tests with MemStore

### Phase 3: Pause API + Resume API

**Goal**: Callers can pause and resume DAG execution via the Go API with correct error handling, CAS semantics, and consistency with the existing Cancel pattern.
**Depends on**: Phase 1, Phase 2
**Plans**: 2 plans

Plans:

- [x] 03-01-PLAN.md — Implement Pause/Resume API functions, sentinel errors, EventResumed, and scheduler handler
- [x] 03-02-PLAN.md — Write comprehensive unit tests for Pause/Resume API

**Details:**

- `Pause(ctx, wf, dagID)` CAS-transitions running→pausing (or running→paused if zero in-flight steps) and returns nil (API-01)
- `Resume(ctx, wf, dagID)` CAS-transitions paused→running, publishes EventResumed for scheduler to re-enqueue ready steps, and returns nil (API-02)
- Both functions return descriptive errors for invalid transitions — e.g., pause on canceled DAG, resume on running DAG (API-03)
- Both functions use KV CAS with retry (5 attempts, ErrStaleRevision on exhaustion), consistent with existing Cancel() pattern (API-04)

### Phase 4: CLI Commands + Integration Tests

**Goal**: Operators can pause and resume DAGs from the CLI with clear output; full end-to-end system passes integration and race-condition tests.
**Depends on**: Phase 3
**Plans**: 3 plans

Plans:

- [x] 04-01-PLAN.md — CLI pause/resume commands + dag ls status filter update
- [x] 04-02-PLAN.md — E2E integration test + race condition tests
- [x] 04-03-PLAN.md — CLI command tests

**Details:**

- `ebctl dag pause <id>` calls DagPause and outputs confirmation with DAG ID (CLI-01)
- `ebctl dag resume <id>` calls DagResume and outputs confirmation with DAG ID (CLI-02)
- `ebctl dag ls` displays PAUSING and PAUSED in the status column alongside RUNNING, DONE, FAILED, CANCELED (CLI-03)
- CLI error messages for invalid pause/resume attempts (e.g., pause on canceled DAG) are clear and actionable (CLI-04)
- E2E integration tests with embedded NATS pass (TST-03)
- Race condition tests for concurrent pause+resume, pause+cancel, and pause+step-completion pass (TST-04)
- CLI command tests pass (TST-05)

## Milestone Summary

**Decimal Phases:** None

**Key Decisions:**

- D-01: Constants use lowercase `"pausing"` and `"paused"` consistent with existing convention
- D-05: `Terminal()` returns `(DAGStatusRunning, false)` for pausing/paused — keeps DAG alive
- D-06/D-07: `Cancel()` transitions both pausing and paused DAGs to canceled
- D-10: `PausedAt` uses `omitempty` for backward compatibility with existing DAGMeta records
- Auto-fix: handleEvent guard allows EventCompleted for pausing DAGs (auto-transition was dead code)
- Auto-fix: Sweep auto-finalize uses inline all-terminal check (state.Terminal() returns false for paused meta)
- Resume publishes EventResumed on bus (not direct enqueue) — through scheduler's serialized path
- No Confirm() for pause/resume CLI — both operations are reversible
- E2E tests use blocking handlers + 3-step DAG for deterministic timing

**Issues Resolved:**

- handleEvent guard blocked EventCompleted for pausing DAGs (auto-transition dead code) — fixed
- Sweep auto-finalize used state.Terminal() which returns false for paused meta — fixed
- Plan's test code used non-existent APIs (task.WithArgs, task.Ref) — corrected
- Plan's 2-step DAG with fast handlers completed before Pause could be called — restructured to blocking handlers + 3-step DAG

**Issues Deferred:** None

**Technical Debt Incurred:** None
