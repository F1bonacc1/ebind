# Requirements Archive: v1.0 — Pause/Resume DAG Execution

**Milestone:** v1.0
**Shipped:** 2026-06-03
**Original Definition:** 2026-06-03

## v1 Requirements — All Completed

### State Machine

- [x] **ST-01**: New DAGStatus constants `DAGStatusPausing` and `DAGStatusPaused` following existing lowercase naming convention — **Validated (Phase 1)**
- [x] **ST-02**: Pure function `HasInFlightSteps(state)` returns true if any step has StatusRunning — **Validated (Phase 1)**
- [x] **ST-03**: Pure function `CanPause(meta)` returns true only if meta.Status is DAGStatusRunning — **Validated (Phase 1)**
- [x] **ST-04**: Pure function `CanResume(meta)` returns true only if meta.Status is DAGStatusPaused — **Validated (Phase 1)**
- [x] **ST-05**: `maybeFinalize` guards against finalizing DAGs in pausing/paused status — **Validated (Phase 1)**
- [x] **ST-06**: `Cancel()` handles pausing/paused states explicitly (CAS transition to canceled) — **Validated (Phase 1)**

### Scheduler Gates

- [x] **SG-01**: Event entry gate — when DAG status is pausing or paused, Ack the completion event without processing (no step transitions, no downstream dispatch) — **Validated (Phase 2)**
- [x] **SG-02**: Dispatch gate — `enqueueReady` checks DAG status is running before enqueuing; skip if pausing/paused — **Validated (Phase 2)**
- [x] **SG-03**: Sweep recovery — on leader acquisition, sweep detects pausing DAGs and transitions them to paused; sweep skips paused DAGs entirely — **Validated (Phase 2)**
- [x] **SG-04**: `pausing→paused` transition occurs automatically when last in-flight step completes (event-driven, no ticker) — **Validated (Phase 2)**

### API Functions

- [x] **API-01**: `Workflow.DagPause(ctx, dagID) error` — CAS transition from running → pausing; if no in-flight steps, skip directly to paused — **Validated (Phase 3)**
- [x] **API-02**: `Workflow.DagResume(ctx, dagID) error` — CAS transition from paused → running; trigger DAG re-evaluation (enqueue ready steps) — **Validated (Phase 3)**
- [x] **API-03**: Both functions return descriptive errors for invalid state transitions (e.g., pause on canceled DAG, resume on running DAG) — **Validated (Phase 3)**
- [x] **API-04**: Both functions use KV CAS with retry, consistent with existing Cancel() pattern — **Validated (Phase 3)**

### CLI Commands

- [x] **CLI-01**: `ebctl dag pause <id>` — calls DagPause, displays confirmation — **Validated (Phase 4)**
- [x] **CLI-02**: `ebctl dag resume <id>` — calls DagResume, displays confirmation — **Validated (Phase 4)**
- [x] **CLI-03**: `ebctl dag ls` — status column displays PAUSING and PAUSED states alongside existing statuses — **Validated (Phase 4)**
- [x] **CLI-04**: Error messages for invalid pause/resume attempts are clear and actionable — **Validated (Phase 4)**

### Testing

- [x] **TST-01**: Pure unit tests for new state machine functions (ST-01 through ST-06) — **Validated (Phase 1)**
- [x] **TST-02**: Scheduler unit tests with MemStore + toggleElector for pause gate behavior (SG-01 through SG-04) — **Validated (Phase 2)**
- [x] **TST-03**: Integration tests with embedded NATS for end-to-end pause/resume — **Validated (Phase 4)**
- [x] **TST-04**: Race condition tests for concurrent pause + resume, pause + cancel, pause + step completion — **Validated (Phase 4)**
- [x] **TST-05**: CLI command tests — **Validated (Phase 4)**
